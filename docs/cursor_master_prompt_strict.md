# CITY GO — MASTER PROMPT FOR CURSOR (STRICT MODE)

Ты работаешь в проекте City Go.

## ОБЯЗАТЕЛЬНЫЕ ПРАВИЛА ОТВЕТА

1. Один ответ = один файл.
2. Всегда присылай полный файл целиком.
3. Никаких diff.
4. Никаких фрагментов.
5. Никаких "найди и замени".
6. Никаких пояснений вне файла.
7. Никакого текста до строки NEW FILE или UPDATED FILE.
8. Никакого текста после полного содержимого файла.
9. Ответ должен быть пригоден для прямого копирования.

## СТРОГИЙ ФОРМАТ ОТВЕТА

Первая строка:

`NEW FILE`  
или  
`UPDATED FILE`

Вторая строка: путь к файлу.

Дальше: полный файл целиком, от первой до последней строки.

Никаких дополнительных комментариев снаружи.  
Никаких списков «что я сделал».  
Никаких объяснений.  
Только один файл за ответ.

## ТРЕБОВАНИЯ К КОДУ

В production-коде обязательно добавляй нормальные комментарии:

- краткий комментарий в начале файла: что это за файл и зачем он нужен;
- комментарий над классом: роль класса;
- комментарий над ключевыми методами: что делает метод;
- комментарии должны быть по делу, без мусора.

После любого нового функционала обязательно:

1. добавить тесты;
2. прогнать тесты;
3. прогнать связанные smoke / regression тесты;
4. прислать команды запуска тестов;
5. прислать результаты прогонов.

## ТРЕБОВАНИЯ К АРХИТЕКТУРНОЙ ДОКУМЕНТАЦИИ

После любых изменений кода ты обязан обновлять архитектурную документацию проекта.

Нужно поддерживать актуальным файл:

`docs/architecture/backend_file_map.md`

В нём для каждого важного backend-файла должно быть:

- путь;
- слой: service / schema / model / router / test / data;
- роль файла;
- шаг pipeline, если есть;
- зависимости;
- что возвращает или производит;
- source of truth это или вспомогательный слой.

Если:

- файл добавлен → добавить его в реестр;
- файл изменён → обновить описание;
- файл удалён → убрать из реестра.

## ПРАВИЛА ДЛЯ ПРОЕКТА CITY GO

1. Не переписывать проект с нуля.
2. Не плодить параллельные архитектуры.
3. Не создавать новый app-layer, если он не существует в реальной структуре проекта.
4. Использовать только реальные файлы и реальные entry points проекта.
5. Не ломать старый `itinerary_*` слой, если задача явно не требует этого.
6. Работать аккуратно и минимально достаточно.
7. После каждого функционала сразу добавлять тесты и прогонять их.
8. Всегда давать полный ответ в формате, пригодном для копирования.

## ОБЯЗАТЕЛЬНЫЙ ШАБЛОН ДЛЯ ЛЮБОЙ ЗАДАЧИ

Когда даётся задача по коду, нужно:

Сначала определить минимально необходимый набор файлов.  
Потом работать строго по формату:

- один ответ = один файл;
- полный файл целиком;
- без внешних пояснений.

Если задача требует нескольких файлов:

- присылать их по одному;
- каждый файл отдельным ответом;
- каждый раз полный файл целиком.

Если задача требует тестов:

- сначала production-файл;
- потом тестовый файл;
- потом обновление архитектурной документации;
- потом команды запуска;
- потом краткий результат прогонов.

## ФОРМАТ ЗАДАЧ ДЛЯ CURSOR

Пример ответа:

    NEW FILE
    path/to/file.py
    <полный код файла целиком>

или

    UPDATED FILE
    path/to/file.py
    <полный код файла целиком>

Без всего остального.

## ЕСЛИ ЗАДАЧА ПРО ТЕСТЫ

Тесты должны:

- покрывать новый функционал;
- покрывать основные сценарии;
- покрывать edge cases;
- не ломать существующий smoke;
- использовать реальную структуру проекта.

**Обязательно после тестов:**

- дать команды запуска;
- указать какие тесты прогнаны;
- указать что зелёное.
