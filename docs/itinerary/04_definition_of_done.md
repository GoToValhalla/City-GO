# City Go — Definition of Done (Itinerary & Replan)

## Зачем нужен этот файл

Этот документ определяет:
- когда фича считается **реально завершённой**, а не “примерно работает”;
- какие условия должны быть выполнены, чтобы считать систему **production-ready**;
- какие тесты и проверки обязательны.

Важно:
- Done = не “код написан”, а “поведение корректно и проверено”;
- DoD применяется к generate, replan и всем связанным слоям.

---

# 1. Общий Definition of Done

Фича считается завершённой, если:

### 1.1 Функциональность
- [ ] реализован весь основной сценарий use case;
- [ ] покрыты альтернативные сценарии;
- [ ] обработаны edge cases;
- [ ] нет критических логических дыр.

### 1.2 Устойчивость
- [ ] нет 500 ошибок на некорректных или неполных данных;
- [ ] все fallback сценарии работают;
- [ ] malformed данные не ломают систему.

### 1.3 Объяснимость
- [ ] система может объяснить свои решения;
- [ ] explanation не пустой;
- [ ] ограничения и деградации явно отражены.

### 1.4 Консистентность
- [ ] generate и replan ведут себя одинаково по ключевым правилам;
- [ ] контекст не теряется между шагами;
- [ ] одинаковые входы дают предсказуемый результат.

### 1.5 Тестирование
- [ ] покрыты соответствующие test cases;
- [ ] high-risk regression tests проходят;
- [ ] поведение проверено на реальных сценариях.

---

# 2. Opening Hours — Definition of Done

## 2.1 Parsing и базовая корректность
- [ ] opening_hours корректно парсятся;
- [ ] malformed данные не вызывают crash;
- [ ] fallback используется при ошибках.

## 2.2 Timezone
- [ ] используется timezone города;
- [ ] при отсутствии timezone используется fallback;
- [ ] fallback логируется и отражается в explanation.

## 2.3 Overlap logic
- [ ] учитывается overlap между visit window и open window;
- [ ] partial overlap даёт сниженный score, а не бинарный результат;
- [ ] нулевой overlap даёт корректное поведение (swap / warning).

## 2.4 Midnight crossing
- [ ] корректно обрабатываются окна через полночь;
- [ ] учитывается окно предыдущего дня;
- [ ] 24h кейс работает.

## 2.5 Candidate pool
- [ ] места не исключаются только потому, что закрыты в момент старта;
- [ ] исключаются только места, закрытые весь день (если данные надёжны).

## 2.6 Pass 2 validation
- [ ] visit_time считается для каждой точки;
- [ ] opening hours проверяются относительно visit_time;
- [ ] некорректные точки не остаются без проверки.

# 3. Route Building — Definition of Done

## 3.1 Порядок маршрута
- [ ] порядок точек логически согласован;
- [ ] маршрут не прыгает хаотично по карте;
- [ ] расстояния между точками разумны.

## 3.2 Time budget
- [ ] маршрут укладывается в time_budget с допустимой погрешностью;
- [ ] хвост корректно обрезается при необходимости;
- [ ] минимальный осмысленный маршрут сохраняется.

## 3.3 Opens later logic
- [ ] места, открывающиеся позже, не теряются преждевременно;
- [ ] такие места могут появляться в поздних позициях маршрута;
- [ ] невалидные позиции отфильтровываются.

## 3.4 Closing soon
- [ ] места, которые закрываются рано, не оказываются в конце маршрута;
- [ ] при необходимости применяется reorder или объяснение.

## 3.5 Waiting gaps
- [ ] ранний приход к месту не игнорируется;
- [ ] small gap допускается с предупреждением;
- [ ] большой gap обрабатывается (swap или warning).

---

# 4. Scoring — Definition of Done

## 4.1 Interest scoring
- [ ] интересы пользователя влияют на результат;
- [ ] нерелевантные точки не доминируют без причины.

## 4.2 Distance scoring
- [ ] близкие точки получают преимущество;
- [ ] при коротком бюджете distance усиливается.

## 4.3 Opening hours scoring
- [ ] OH используется как модификатор, а не единственный фактор;
- [ ] отсутствие данных не убивает точку.

## 4.4 Diversity
- [ ] маршрут не состоит из однотипных точек без запроса;
- [ ] diversity учитывается на уровне всего маршрута.

---

# 5. Replan — Definition of Done

## 5.1 Контекст
- [ ] current_route_context содержит все ключевые параметры;
- [ ] параметры не теряются между запросами;
- [ ] budget, режимы и ограничения сохраняются.

## 5.2 Coffee stop
- [ ] вставляется ближайшая подходящая точка;
- [ ] точка открыта в момент вставки;
- [ ] не нарушает budget.

## 5.3 Shorten route
- [ ] маршрут корректно сокращается;
- [ ] остаётся валидным.

## 5.4 Continue from here
- [ ] маршрут перестраивается от новой позиции;
- [ ] порядок адаптируется к новому старту.

## 5.5 Multi-stop replan
- [ ] completed точки не возвращаются;
- [ ] остаток маршрута пересобирается корректно.

## 5.6 Failure-safe replan
- [ ] отсутствие подходящих точек не ломает маршрут;
- [ ] система возвращает осмысленный fallback.

---

# 6. Failure Handling — Definition of Done

## 6.1 No places
- [ ] возвращается валидный пустой ответ;
- [ ] explanation объясняет ситуацию.

## 6.2 All closed
- [ ] система не притворяется, что всё открыто;
- [ ] предупреждения присутствуют.

## 6.3 Missing data
- [ ] система работает при отсутствии данных;
- [ ] пользователь информирован о неполноте данных.

## 6.4 Malformed data
- [ ] система не падает;
- [ ] используются fallback значения;
- [ ] ошибки логируются.

## 6.5 Reserve exhausted
- [ ] swap невозможен → маршрут остаётся;
- [ ] warning отображается.

---

# 7. Explanation — Definition of Done

## 7.1 Point-level
- [ ] у каждой точки есть reason;
- [ ] reason объясняет:
  - релевантность;
  - timing;
  - ограничения;
  - warnings.

## 7.2 Route-level
- [ ] explanation всегда заполнен;
- [ ] включает:
  - общее время;
  - ограничения;
  - деградации;
  - проблемы данных.

## 7.3 Transparency
- [ ] система не скрывает uncertainty;
- [ ] пользователь понимает ограничения маршрута.

---

# 8. Data Quality — Definition of Done

## 8.1 Opening hours
- [ ] система работает при частичном покрытии;
- [ ] fallback корректен.

## 8.2 Timezone
- [ ] fallback не ломает расчёты;
- [ ] отражается в логах и explanation.

## 8.3 Missing fields
- [ ] отсутствующие поля не приводят к crash;
- [ ] используются разумные defaults.

---

# 9. Performance — Definition of Done

- [ ] генерация маршрута < 2 секунд;
- [ ] replan < 1.5 секунд;
- [ ] нет экспоненциального роста времени;
- [ ] нет зацикливаний в pass 2.

---

# 10. Regression Safety

Перед деплоем должны проходить:

- [ ] high-risk regression тесты (см. test_matrix);
- [ ] opening-hours критические сценарии;
- [ ] replan сценарии;
- [ ] edge cases (empty, missing data, malformed data).

---

# 11. Продуктовый принцип Done

Фича считается готовой, если:

> Маршрут не только “работает”,
> но и:
> - логичен,
> - предсказуем,
> - объясним,
> - не вводит пользователя в заблуждение.

Если есть выбор между:
- “умным, но непрозрачным”
- и “простым, но честным”

→ выбираем второе.
