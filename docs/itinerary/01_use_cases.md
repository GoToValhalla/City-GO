# City Go — Itinerary Use Cases

## Зачем нужен этот файл

Этот документ фиксирует **продуктовые и системные use cases** для itinerary/replan блока City Go.

Он нужен как:
- единый source of truth по сценариям;
- основа для backend-логики;
- база для QA-покрытия;
- основа для дальнейшей декомпозиции в roadmap, test matrix и status tracking.

Важно:
- здесь **нет статусов реализации**;
- здесь **нет предположений**, что уже сделано;
- здесь фиксируется только то, **как система должна себя вести**.
- Сопоставление с кодом (itinerary vs `POST /recommendations/route`): [`08_implementation_coverage.md`](08_implementation_coverage.md).

---

## Общие принципы

City Go строит не просто список мест, а **маршрут во времени**.

Это значит:
- каждое место оценивается в контексте предполагаемого времени посещения;
- порядок мест влияет на валидность маршрута;
- часы работы мест, время на дорогу и длительность визита — часть ядра логики;
- replan — это не отдельный костыль, а продолжение той же модели маршрута.

# 1. Core Use Cases

## UC-001 — Generate itinerary from scratch
**Категория:** Core  
**Приоритет:** MVP

### Description
Пользователь хочет получить готовый маршрут в незнакомом городе по интересам, времени и базовым ограничениям.

### User Input
- `city_slug`
- `trip_start_datetime` или без него
- `time_budget_minutes` или open duration режим
- `route_mode`
- интересы / текстовый запрос
- дополнительные ограничения

### Preconditions
- город существует в системе;
- в городе есть активные точки;
- start context может быть определён явно или через fallback.

### Main Flow
1. Система определяет `start_context`.
2. Система получает candidate places.
3. Применяет только реально обязательные ограничения.
4. Ранжирует кандидатов.
5. Собирает упорядоченный маршрут.
6. Проверяет маршрут по времени.
7. Добавляет explanation.
8. Возвращает current route context.

### Alternative Flows
- нет `trip_start_datetime` → система работает в degraded режиме;
- нет явного старта → используется city anchor;
- кандидатов мало → система расширяет радиус поиска.

### Edge Cases
- в городе слишком мало мест;
- opening hours заполнены частично;
- start context неточный;
- time budget слишком короткий.

### Expected Result
- осмысленный упорядоченный маршрут;
- оценка длительности и дистанции;
- explanation на уровне маршрута и точек;
- current route context для последующего replan.

### Acceptance Criteria
- [ ] маршрут не пустой, если в городе есть релевантные места;
- [ ] точки упорядочены логически и географически;
- [ ] explanation заполнен;
- [ ] current route context возвращается;
- [ ] система не падает при неполных данных.

## UC-002 — Quick walk near me
**Категория:** Core  
**Приоритет:** MVP

### Description
Пользователь хочет короткий маршрут рядом с текущей локацией, без сложного планирования.

### User Input
- `current_location`
- `time_budget_minutes`
- `route_mode = walk`

### Preconditions
- есть GPS-координаты пользователя;
- можно определить город или ближайший городской контекст.

### Main Flow
1. Система определяет старт из GPS.
2. Берёт компактный радиус поиска.
3. Ранжирует ближайшие релевантные точки.
4. Собирает короткий walk-маршрут.

### Alternative Flows
- плохая точность GPS → радиус увеличивается;
- рядом нет мест → поиск расширяется.

### Edge Cases
- пользователь на границе города;
- пользователь в зоне без POI;
- короткий бюджет 15–30 минут.

### Expected Result
- 1–3 точки рядом;
- минимальные переходы;
- акцент на расстоянии и доступности.

### Acceptance Criteria
- [ ] маршрут начинается рядом со стартом;
- [ ] переходы короткие;
- [ ] маршрут укладывается в короткий бюджет;
- [ ] explanation отражает ограничения.

## UC-003 — Full day itinerary
**Категория:** Core  
**Приоритет:** MVP

### Description
Пользователь хочет маршрут на большую часть дня с несколькими типами точек и нормальным темпом.

### User Input
- `trip_start_datetime`
- `time_budget_minutes = 360–480+`
- широкий набор интересов
- `max_places`

### Preconditions
- в городе достаточно разных категорий мест;
- доступны long-duration сценарии.

### Main Flow
1. Система берёт широкий пул кандидатов.
2. Строит разнообразный маршрут.
3. Учитывает временное окно маршрута.
4. При необходимости вставляет логичные типы точек по ходу дня.
5. Валидирует результат по времени и доступности.

### Alternative Flows
- в городе мало разнообразия → маршрут строится из того, что есть;
- нет food/cafe категорий → система не падает, а честно это отражает.

### Edge Cases
- маршрут пересекает обеденное время;
- часть точек закрывается раньше середины маршрута;
- вторая половина дня бедна на доступные точки.

### Expected Result
- длинный, но не хаотичный маршрут;
- несколько типов точек;
- нормальный ритм.

### Acceptance Criteria
- [ ] в длинном маршруте есть сценарное разнообразие;
- [ ] маршрут не разваливается из-за часов работы;
- [ ] explanation отражает важные ограничения;
- [ ] маршрут не перегружен однотипными точками.

## UC-004 — Interest-driven route
**Категория:** Core  
**Приоритет:** MVP

### Description
Пользователь хочет маршрут под конкретный интерес или узкий сценарий.

### User Input
- интересы или текстовый запрос вида:
  - кофе
  - музеи
  - прогулка у моря
  - архитектура

### Preconditions
- в городе есть места соответствующего типа.

### Main Flow
1. Система усиливает interest matching.
2. Снижает роль ненужного разнообразия.
3. Строит маршрут преимущественно из релевантных категорий.

### Alternative Flows
- релевантных точек мало → система возвращает ограниченный маршрут и объясняет это.

### Edge Cases
- пользователь хочет только один тип мест;
- все релевантные места закрыты;
- релевантных мест мало.

### Expected Result
- маршрут под конкретный сценарий, а не “сборная солянка”.

### Acceptance Criteria
- [ ] большая часть маршрута соответствует запросу;
- [ ] лишние категории не доминируют;
- [ ] ограничения честно отражены в explanation.

# 2. Context-Aware Use Cases

## UC-010 — Route with dog
**Категория:** Context-aware  
**Приоритет:** MVP

### Description
Пользователь идёт с собакой и хочет маршрут только через подходящие места.

### User Input
- `with_dog = true`

### Preconditions
- часть мест размечена по dog-friendly признакам.

### Main Flow
1. Кандидаты фильтруются или сильно понижаются по dog-unfriendly признаку.
2. Парки и dog-friendly точки получают приоритет.
3. Маршрут строится в dog-safe логике.

### Edge Cases
- dog-friendly данных мало или нет;
- кафе dog-friendly по факту, но не размечены;
- в городе почти нет dog-friendly indoor точек.

### Expected Result
- маршрут не ведёт пользователя в заведомо неудобные места.

### Acceptance Criteria
- [ ] в выходном маршруте нет явных dog-unfriendly мест;
- [ ] explanation отражает dog context;
- [ ] при отсутствии dog-friendly мест система деградирует честно.

## UC-011 — Budget-conscious route
**Категория:** Context-aware  
**Приоритет:** MVP

### Description
Пользователь хочет маршрут под ограниченный бюджет.

### User Input
- `budget_level`

### Preconditions
- у мест есть price metadata или хотя бы частично заполненный `price_level`.

### Main Flow
1. Дорогие места фильтруются или штрафуются.
2. Дешёвые и бесплатные точки получают преимущество.
3. При replan бюджет должен сохраняться и переиспользоваться.

### Edge Cases
- `price_level` отсутствует у части мест;
- дешёвых точек мало;
- replan пытается вставить дорогой stop.

### Expected Result
- маршрут соответствует бюджету настолько, насколько позволяют данные.

### Acceptance Criteria
- [ ] budget влияет на generate;
- [ ] budget не теряется при replan;
- [ ] explanation отражает ограничения по цене;
- [ ] система не притворяется, что знает точные цены, если данных нет.

## UC-013 — Anti-tourist mode
**Категория:** Context-aware  
**Приоритет:** V1

### Description
Пользователь хочет менее туристический и более локальный маршрут.

### User Input
- `anti_tourist = true`

### Preconditions
- есть хотя бы proxy-сигналы туристичности / локальности.

### Main Flow
1. Туристические точки штрафуются.
2. Тихие / локальные точки получают бонус.
3. Это soft scoring, а не абсолютный запрет.

### Edge Cases
- в городе почти все точки туристические;
- популярность данных нет;
- anti-tourist конфликтует с must-see логикой.

### Expected Result
- маршрут не ведёт пользователя по самым очевидным spot’ам без причины.

### Acceptance Criteria
- [ ] anti-tourist влияет на ranking;
- [ ] маршрут не становится пустым только из-за anti-tourist;
- [ ] explanation отражает этот режим.

# 3. Time-Aware Use Cases

## UC-020 — Evening route
**Категория:** Time-aware  
**Приоритет:** MVP

### Description
Маршрут начинается вечером, и система должна уметь работать с точками, которые открыты позже или закрываются рано.

### User Input
- `trip_start_datetime = evening`
- `time_budget_minutes`

### Main Flow
1. На retrieval этапе исключаются только места, закрытые весь день, а не просто закрытые в момент старта.
2. Pass 1 учитывает вечерний старт.
3. Pass 2 валидирует точки по реальному `visit_time`.

### Edge Cases
- точка открывается позже старта;
- точка закрывается до прихода пользователя;
- почти все дневные места уже недоступны.

### Expected Result
- вечерний маршрут не должен терять хорошие точки, которые откроются позже по ходу маршрута.

### Acceptance Criteria
- [ ] места, открывающиеся позже старта, могут остаться в candidate pool;
- [ ] закрывающиеся слишком рано точки не попадают в неверную позицию;
- [ ] explanation отражает вечерние ограничения.

---

## UC-021 — Closing soon prioritization
**Категория:** Time-aware  
**Приоритет:** V1

### Description
Если место закрывается заметно раньше других, система должна уметь поднять его выше по маршруту.

### User Input
- `trip_start_datetime`
- точки с разным closing time

### Main Flow
1. Система вычисляет `minutes_until_close` для предполагаемого времени визита.
2. Если место скоро закрывается и ещё имеет смысл идти, оно получает urgency bonus.
3. В нужных случаях применяется локальный reorder.

### Edge Cases
- место закрывается рано, но слишком далеко;
- несколько точек закрываются скоро;
- reorder ломает геометрию маршрута.

### Expected Result
- маршрут не должен приводить пользователя к уже закрытому месту, если можно было переставить порядок.

### Acceptance Criteria
- [ ] “closing soon” влияет на позицию в маршруте;
- [ ] reorder не ломает маршрут чрезмерно;
- [ ] причина перестановки отражается в explanation.

## UC-022 — Midnight crossing
**Категория:** Time-aware  
**Приоритет:** MVP

### Description
Система должна корректно работать с местами, которые открыты через полночь.

### User Input
- `trip_start_datetime = late evening/night`
- place with `open > close` in same calendar day format

### Main Flow
1. Opening window нормализуется.
2. Система учитывает окно предыдущего дня, если оно тянется за полночь.
3. `oh_score` считается корректно.

### Edge Cases
- 24h точки;
- визит после полуночи;
- визит уже после закрытия;
- место открывается в 23:00, визит в 00:30.

### Expected Result
- ночные маршруты не ломаются на midnight crossing.

### Acceptance Criteria
- [ ] overnight окна поддерживаются;
- [ ] вчерашнее окно учитывается, если нужно;
- [ ] 24h case обрабатывается корректно.

## UC-023 — Waiting gap scenario
**Категория:** Time-aware  
**Приоритет:** V1

### Description
Пользователь приходит к месту раньше открытия. Система должна решить: подождать, переставить или заменить.

### User Input
- `visit_time < open_time`

### Main Flow
1. Вычисляется `gap_minutes`.
2. Если gap маленький — точка может быть оставлена с warning.
3. Если gap большой и есть хорошие альтернативы — делается swap.
4. Если альтернатив нет — маршрут остаётся, но с честным предупреждением.

### Edge Cases
- gap 5 минут;
- gap 30 минут;
- gap 90 минут;
- альтернатив нет.

### Expected Result
- система не делает вид, что точка доступна прямо сейчас, если это не так.

### Acceptance Criteria
- [ ] маленький wait gap может быть допустим;
- [ ] большой gap при наличии альтернатив должен убираться;
- [ ] warning должен объяснять ожидание.

---

## UC-024 — Route truncation due to time budget
**Категория:** Time-aware  
**Приоритет:** MVP

### Description
Если итоговый маршрут не помещается в бюджет, хвост должен быть сокращён.

### User Input
- `time_budget_minutes`

### Main Flow
1. Строится маршрут-кандидат.
2. Считается длительность.
3. Если маршрут слишком длинный, последние точки обрезаются.
4. Система сохраняет минимально осмысленный маршрут.

### Edge Cases
- даже 1 точка почти не помещается;
- бюджет экстремально короткий;
- почти весь маршрут обрезан.

### Expected Result
- система не возвращает маршрут, заведомо не помещающийся по времени.

### Acceptance Criteria
- [ ] маршрут после trim укладывается в бюджет с допустимой толерантностью;
- [ ] хвост режется, а не середина;
- [ ] explanation отражает факт сокращения.

## UC-025 — Place opens later during route
**Категория:** Time-aware  
**Приоритет:** MVP

### Description
Если маршрут начинается до открытия точки, но эта точка откроется до конца маршрута, система должна уметь использовать её позже по порядку.

### Пример
- `trip_start = 15:00`
- `time_budget = 4h`
- место открывается в `18:00`

### Main Flow
1. Точка не исключается на retrieval этапе, если не закрыта весь день.
2. Она участвует в ranking.
3. На ранних позициях она должна получать плохой `oh_score`.
4. На поздней позиции она может стать валидной.
5. Если подходящей позиции нет, точка должна уйти из маршрута.

### Edge Cases
- точка открывается позже старта, но раньше конца маршрута;
- точка открывается уже после окончания маршрута;
- точка оказывается в wrong position и должна быть заменена.

### Expected Result
- система не теряет такие точки слишком рано;
- и не ставит их туда, где они ещё недоступны.

### Acceptance Criteria
- [ ] точка не фильтруется преждевременно;
- [ ] точка может появиться в поздней части маршрута;
- [ ] если валидной позиции нет, она убирается честно.

---

## UC-026 — Start too late for meaningful route
**Категория:** Time-aware  
**Приоритет:** V1

### Description
Пользователь начинает маршрут слишком поздно, и почти ничего уже не доступно.

### Expected Result
- не строить фальшиво “нормальный” маршрут;
- честно предложить:
  - короткий вечерний маршрут;
  - 24h / viewpoint / walk fallback;
  - или ограниченный набор точек.

### Acceptance Criteria
- [ ] explanation честно сообщает о позднем старте;
- [ ] система не маскирует нехватку доступных точек.

## UC-031 — Loop route
**Категория:** Location-aware  
**Приоритет:** V1

### Description
Пользователь хочет закончить маршрут рядом со стартом.

### User Input
- `return_to_start = true`

### Main Flow
1. Система учитывает возвратную форму маршрута.
2. Последние точки штрафуются, если уводят слишком далеко от старта.
3. Возвратный сегмент учитывается в time/distance.

# 9. Специальные продуктовые правила

## Правило 1 — Place opens later during route
Если:
- маршрут строится с 15:00;
- бюджет 4 часа;
- место открывается в 18:00;

то место:
- **должно попадать в candidate pool**, если не закрыто весь день;
- **должно участвовать в ranking**;
- **может попасть в маршрут**, если для него находится валидная поздняя позиция;
- **не должно ставиться рано**, если тогда оно закрыто;
- **должно исключаться**, если до конца маршрута не успевает открыться вовремя.

## Правило 2 — Opening hours не должны доминировать полностью
Opening hours — важный, но не единственный сигнал.  
Места с длинным графиком не должны автоматически вытеснять более релевантные точки только из-за часов работы.

## Правило 3 — Если данных нет, система деградирует честно
Если:
- нет opening hours,
- нет timezone,
- нет price,
- нет уверенности по start context,

то маршрут всё равно может строиться, но explanation обязан это отражать.
