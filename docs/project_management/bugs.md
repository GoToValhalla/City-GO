# City Go — Баги

Источник истины для актуальных багов проекта City Go.

## Правила ведения

- Новые баги добавлять в конец соответствующего раздела.
- Не удалять баг без явного решения.
- После исправления менять статус на `Fixed`.
- Если баг не подтвердился, менять статус на `Rejected` и указать причину.
- Если баг перенесен в задачу/issue, добавлять ссылку или номер задачи.

## Статусы

- `Open` — актуально, не исправлено.
- `In Progress` — взято в работу.
- `Fixed` — исправлено.
- `Rejected` — не подтвердилось или не является багом.
- `Deferred` — отложено.

## Critical / High

### BUG-001 — Нет фото мест

Статус: Open  
Приоритет: High  
Компонент: Данные / Places / Images

Описание:  
У мест отсутствуют фотографии.

Ожидаемый результат:  
У мест должны отображаться реальные фотографии или корректные fallback-изображения по категории.

Фактический результат:  
Фото мест отсутствуют.

Заметки:  
Нужно проверить pipeline импорта/обогащения мест, сохранение изображений и frontend-отображение.

---

### BUG-002 — У части мест нет адреса

Статус: Open  
Приоритет: High  
Компонент: Данные / Places / Addresses

Описание:  
У части мест отсутствует адрес.

Ожидаемый результат:  
Для всех мест, где адрес можно определить, должен быть заполнен корректный адрес.

Фактический результат:  
У части мест адрес отсутствует.

Заметки:  
Нужно проверить импорт OSM-тегов, backfill адресов, запуск cron/pipeline и отображение адреса на frontend.

### BUG-003 — Нет возможности посмотреть все места

Статус: Open  
Приоритет: High  
Компонент: Places / UI / API

Описание:  
Нет возможности посмотреть все места. Пользователю доступны только предзагруженные места.

Ожидаемый результат:  
Пользователь должен иметь возможность открыть полный список/каталог доступных мест по выбранному городу.

Фактический результат:  
Доступны только предзагруженные места.

Заметки:  
Нужно проверить существующие endpoints, frontend-экран мест, фильтрацию по городу и ограничения текущей выдачи.

---

### BUG-004 — Белая страница при вводе букв в поле поиска

Статус: Open  
Приоритет: High  
Компонент: Search / UI / Frontend

Описание:  
При клике на поле поиска и попытке ввода буквенных символов появляется белая страница.

Шаги воспроизведения:  
1. Открыть приложение.  
2. Кликнуть в поле поиска.  
3. Начать вводить буквы.

Ожидаемый результат:  
Поле поиска принимает буквенный ввод, страница не падает, пользователь видит результаты поиска, пустое состояние или сообщение об отсутствии результатов.

Фактический результат:  
Появляется белая страница.

Временный обход:  
Помогает обновление страницы.

Заметки:  
Нужно проверить frontend-ошибку в консоли, обработку search query, состояние результатов, debounce/фильтрацию и обработку пустых/нечисловых значений.

### BUG-005 — Аптеки попадают в прогулочные маршруты

Статус: Open  
Приоритет: High  
Компонент: Routes / Recommendation / Category Filtering

Описание:  
В прогулочный маршрут попадают аптеки и медицинские места. На скриншоте в маршруте отображаются ValeoFarm и Фармаимпекс.

Ожидаемый результат:  
Аптеки, медицинские точки и сервисные места не должны попадать в туристический/прогулочный маршрут по умолчанию. Такие категории допустимы только в специальных сценариях вроде “аптека рядом”, “медицина”, “срочно нужно купить лекарство”.

Фактический результат:  
Аптеки отображаются как точки маршрута и получают объяснение “логично ложится в порядок прогулки рядом с предыдущей точкой”.

Заметки:  
Нужно проверить category filtering, route scenario filtering, scoring boosts, fallback-кандидаты и seed/category mapping. Медицинские категории должны быть hard-excluded из обычных прогулочных маршрутов.

### BUG-006 — Карточки маршрута выглядят как технический черновик

Статус: Open  
Приоритет: High  
Компонент: Routes / UI / UX

Описание:  
Карточки точек маршрута выглядят как технический черновик: placeholder “Место”, крупные технические тексты, координаты вместо нормального адреса, слабая визуальная иерархия, неготовый вид для пользователя.

Ожидаемый результат:  
Карточка маршрута должна выглядеть как готовый пользовательский интерфейс: фото/корректный placeholder, название, тип места, адрес или понятная локация, время, длительность визита, причина выбора нормальным человеческим текстом, аккуратная визуальная структура.

Фактический результат:  
Пользователь видит карточку, похожую на debug/demo UI, включая координаты и технические формулировки.

Заметки:  
Нужно переделать frontend-компонент route point card, скрыть технические поля из пользовательского режима, привести reason text к нормальному виду, проверить mobile layout и empty states.
