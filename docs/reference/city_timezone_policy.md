# City Go — City Timezone Policy

## Зачем нужен этот документ

Таймзона — критически важная часть логики маршрутов:

- opening_hours считаются относительно локального времени города
- visit_time рассчитывается в контексте timezone
- ошибки в timezone → полностью неправильные маршруты

Этот документ фиксирует правила работы с таймзонами в системе.

---

# 1. Источник правды

## 1.1 Единственный источник истины

Таймзона хранится в модели City:

`City.timezone: str`

Формат:
- строго IANA timezone

Примеры:
- Europe/Berlin
- Europe/Paris
- Europe/Kaliningrad
- Europe/Amsterdam

---

## 1.2 Запрещено

- ❌ вычислять timezone “на лету”
- ❌ определять timezone по стране
- ❌ использовать серверное время
- ❌ хардкодить UTC для всех городов

---

# 2. Требования к данным

## 2.1 Обязательные поля города

Каждый город обязан иметь:

- slug
- name
- country
- timezone

Если timezone отсутствует:
→ город считается невалидным для маршрутов

---

## 2.2 Формат

- только строка IANA
- регистр важен

Не допускаются:
- GMT+2
- UTC+1
- CET

---

# 3. Использование в системе

## 3.1 Generate

При генерации маршрута:

- trip_start интерпретируется в timezone города
- visit_time считается в timezone города
- opening_hours проверяются в timezone города

---

## 3.2 Replan

- используется тот же timezone, что и у исходного маршрута
- timezone не должен теряться в current_route_context

---

## 3.3 Opening hours

Все функции вида:

- is_open_at(...)
- get_open_intervals(...)
- get_minutes_until_close(...)

должны использовать city.timezone

---

# 4. Fallback стратегия

## 4.1 Если timezone отсутствует

Система должна:

- использовать fallback (например UTC)
- пометить маршрут как degraded

---

## 4.2 Explanation (обязательно)

Пользователь должен получить:

"Часть данных о времени может быть неточной (таймзона города не определена)"

---

## 4.3 Логирование

- ошибка должна логироваться
- должна быть возможность найти такие города

---

# 5. Data governance

## 5.1 Seed данные

Все города должны задаваться с timezone в:

data/cities_seed.json

---

## 5.2 При добавлении нового города

Обязательно:

- указать timezone
- проверить корректность (валидный IANA)

---

## 5.3 Проверка

Рекомендуется:

- валидировать timezone через zoneinfo или pytz
- падать на этапе импорта, если timezone некорректен

---

# 6. Edge cases

## 6.1 DST (летнее время)

- используется IANA → DST обрабатывается автоматически
- не требуется ручная логика

---

## 6.2 Нестандартные таймзоны

Система должна поддерживать:

- UTC+5:30
- UTC+9:30

(это покрывается IANA)

---

# 7. QA чек-лист

- [ ] у всех городов есть timezone
- [ ] timezone валиден
- [ ] маршруты считаются в локальном времени
- [ ] opening_hours корректно учитывают timezone
- [ ] fallback работает
- [ ] explanation содержит предупреждение

---

# 8. Definition of Done

Timezone считается реализованным корректно, если:

- ни один маршрут не зависит от server time
- все расчёты идут через city.timezone
- fallback не ломает систему
- пользователь уведомлён о проблемах

---

# Ключевой принцип

Timezone — это не деталь реализации, а часть бизнес-логики маршрута.
